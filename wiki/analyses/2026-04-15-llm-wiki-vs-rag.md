# LLM Wiki vs RAG 비교

## 핵심 차이: 지식의 축적 방식

RAG와 LLM Wiki 패턴은 모두 LLM이 문서를 기반으로 질문에 답하도록 돕지만, 그 구조적 접근 방식이 근본적으로 다르다.

### RAG (Retrieval-Augmented Generation)

- **작동 방식**: 질문이 들어올 때마다 원본 문서 컬렉션에서 관련 청크(chunk)를 검색(retrieve)하고, 그 결과를 LLM에 넘겨 답을 생성한다.
- **지식 상태**: 원본 문서는 그대로 남아 있고, 지식은 매 질의마다 새로 재발견(re-derived)된다.
- **누적 없음**: 이전 질의에서 도출한 합성(synthesis)이나 교차 참조(cross-reference)가 보존되지 않는다. 같은 질문을 10번 해도 LLM은 매번 처음부터 파편들을 짜 맞춰야 한다.
- **대표 시스템**: NotebookLM, ChatGPT 파일 업로드, 대부분의 RAG 파이프라인.

### LLM Wiki 패턴

- **작동 방식**: 새 소스를 추가할 때마다 LLM이 소스를 읽고 기존 wiki에 지식을 통합(integrate)한다. 엔티티 페이지를 업데이트하고, 모순을 표시하며, 합성을 강화한다.
- **지식 상태**: wiki는 영속적·복합적(persistent, compounding) 아티팩트다. 교차 참조는 이미 만들어져 있고, 모순은 이미 표시되어 있으며, 합성은 지금까지 읽은 모든 소스를 반영한다.
- **누적 있음**: 질의 결과(비교표, 분석, 발견한 연결 등)도 wiki 페이지로 저장할 수 있어, 탐색 자체가 지식 베이스를 풍부하게 만든다.
- **구조**: Raw Sources Layer (불변 원본) → Wiki Layer (LLM이 유지하는 마크다운 페이지들) → Schema Layer (CLAUDE.md / AGENTS.md, 동작 규칙 정의)

## 비교 요약

| 차원 | RAG | LLM Wiki |
|---|---|---|
| 지식 통합 시점 | 질의 시점 (매번) | 인제스트 시점 (한 번, 이후 점진적 업데이트) |
| 지식 영속성 | 없음 (원본 문서만 보존) | 있음 (wiki가 합성 결과를 보존) |
| 지식 복합성 | 없음 (매 질의 독립적) | 있음 (소스가 쌓일수록 wiki가 풍부해짐) |
| 교차 참조 | 질의 시마다 재생성 | 인제스트 때 이미 구축됨 |
| 모순 감지 | 질의마다 우연에 의존 | Lint 오퍼레이션으로 주기적 점검 |
| 인프라 복잡도 | 임베딩, 벡터 DB 필요 | 마크다운 파일 + index.md (소규모에서 충분) |
| 답변 재사용 | 채팅 히스토리에 소멸 | wiki 페이지로 저장, 이후 질의에서 참조 가능 |

## 핵심 통찰

LLM Wiki와 RAG는 상호 배타적이지 않다. LLM Wiki는 wiki 규모가 커지면 wiki 내부 검색에 임베딩 기반 검색(예: qmd)을 선택적으로 도입할 수 있다. 그러나 근본적 차이는 **"지식을 언제 합성하느냐"** 에 있다.

- RAG: 질의할 때 합성 → 합성 결과가 사라짐
- LLM Wiki: 인제스트할 때 합성 → 합성 결과가 wiki에 쌓임

이 차이가 Karpathy의 원문에서 말하는 "compounding knowledge"의 본질이다. RAG는 문서를 검색 가능하게 만들지만, LLM Wiki는 문서에서 지식을 추출하여 영속적 구조물로 변환한다.
