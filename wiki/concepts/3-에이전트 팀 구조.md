# 3-에이전트 팀 구조 (Three-Man Team)

단일 에이전트의 역할 이탈과 불필요한 탐색을 막기 위해 **아키텍트 → 빌더 → 리뷰어**로 역할을 분리하는 패턴.

20년 경력 시니어 엔지니어 Russell이 실제 SaaS 프로덕션 환경에서 검증.

## 역할 정의

| 에이전트 | 역할 | 핵심 제약 |
|---|---|---|
| **아키텍트** | 전체 시스템 이해, 작업 계획·브리프 작성, 수정할 파일 목록 지정 | — |
| **빌더** | 아키텍트 브리프만 구현 | **브리프에 없는 작업 절대 금지** |
| **리뷰어** | 완성 코드 검증, 문제 시 구체적 피드백으로 반려 | 변경 부분만 확인 |

## 워크플로우

```
아키텍트 (계획·브리프) → 빌더 (구현) → 리뷰어 (검증) → 아키텍트 (최종 확인)
```

각 단계마다 명확한 인수인계 문서 전달. 어떤 단계도 건너뛸 수 없음.

## 토큰 절감 원리

- 빌더: 전체 코드베이스 읽기 불필요. 아키텍트가 지정한 파일만 수정
- 리뷰어: 변경된 부분만 확인
- 범위 이탈(요청하지 않은 기능 추가, CSS 변경 등) 원천 차단

## 설정

오픈소스 템플릿: [russelleNVy/three-man-team](https://github.com/russelleNVy/three-man-team)

Claude Code에서 "아키텍트로 시작해 달라"고 말하면 팀 구조 자동 활성화.

## 하네스 엔지니어링과의 관계

[[concepts/하네스 엔지니어링]]의 "구조적으로 실수가 불가능하게 만들기" 철학을 **에이전트 역할 분리**로 구현한 형태다. Auto Enforcement System이 코드 품질을 자동 교정하듯, 3-에이전트 구조는 작업 범위를 자동 제한한다.
